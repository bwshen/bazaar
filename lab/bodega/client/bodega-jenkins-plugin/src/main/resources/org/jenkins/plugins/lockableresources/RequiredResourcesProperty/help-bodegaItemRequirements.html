Bodega item requirements JSON/YAML. If specified, this overrides all the other lockable resources settings below. Example:
  pod:
    type: rktest_yml
    requirements:
      location: COLO

If an item named _jenkins_node is included, then it will be used as the Jenkins node for running the job, overriding any node label parameter (SLAVE_LABEL) or assigned node label expression. For example:
  _jenkins_node:
    type: sd_dev_machine
    requirements:
      location: COLO
      version: 347ae6cf7cd3abce074eb9ce6dc2243f5a1aa528

Note that even this mechanism is not the long term goal because values such as the sd_dev_machine version and CDM node version are far more dynamic than the Jenkins job configurations. To really achieve pipeline as code, we would like to discover these values dynamically, just before starting the job, from:

  • The .sd_dev_bootstrap_hash file for the source tree that's about to be built or Rubrik tarball that's about to be tested.
  • The version_number file from the Rubrik tarball that's about to be tested.
  • Declarations of other required test resources from the test code corresponding to the Rubrik tarball that's about to be tested.

If the given item requirements contains the string ${ARTIFACTS_URL_TO_USE} and the ARTIFACTS_URL_TO_USE parameter is specified, then ${ARTIFACTS_URL_TO_USE} in the requirements will be dynamically replaced with the value of the parameter. Note that this behavior is subject to change if we can think of a cleaner and more intuitive solution.
